This document explains the full research process behind the final competitor

PART 1 — WHERE IT STARTED: THE ALGORITHM
--------------------------------------------------------------------------------

The project began with a working algorithm, published here:

  https://camussicamusica.github.io/automated-stalker

This tool was built to scrape and score potential competitors for a given
YouTube creator. It pulled channel data, estimated subscriber ranges, analyzed
topic overlap, and ranked candidates by relevance. On paper, it was solid.

The core assumption baked into that algorithm was:
  > "If someone is a YouTuber talking about marketing and AI, they are a
    viable competitor and probably have a business behind them."

That assumption turned out to be wrong — or at least, wildly incomplete.

When the algorithm ran and returned a list of candidates for Nuno, the results
looked great on the surface: YouTubers, decent subscriber counts, similar
content topics. But when each candidate was manually checked, the same
problem appeared again and again:

  - Organic-only creators. No Facebook ads. No sales funnel. No product.
  - YouTube hobbyists with knowledge but no commercial operation behind them.
  - Educators with courses but zero paid traffic, meaning their acquisition
    model was completely different from Nuno's.

Nuno is not just a YouTuber. He runs Facebook and Instagram ads. He has active
sales funnels. He sells high-ticket programs. His business is built on paid
acquisition feeding into a conversion funnel. His competitors need to match
that profile — not just overlap on content topics.

The algorithm had found the right industry. It had found the wrong type of
player within it.

The logical next step was: update the algorithm to filter for the right signals.

ATTEMPT 2 — Competitor Repo Iterations (02:20 – 04:07)
  Heavy commit activity on the Nuno's Competitors git repo, with presentation

ATTEMPT 3 — Scraping Scripts (19:32 – 20:32)
  By evening, several Python scripts were being written to automate the search:

    nuno_cleaner.py     — Script to clean and normalize Nuno's data
    scrap_compet.py     — Competitor scraping script (two versions: in repo
                          and on Desktop root)
    yt_cleaner.py       — YouTube data cleaning utility
    fb_scrap.py         — Facebook ads scraping attempt

  The fb_scrap.py was the critical one. Facebook's ad library does not expose
  a clean API for this kind of scraping. It requires authentication, handles
  bot detection aggressively, and the data structure changes frequently.
  fb_scrap.py produced errors, partial data, and eventually nothing useful.

  Also generated during this window:
    nuno_motherdoc.txt     — Consolidated notes on Nuno's full ecosystem
    top3_intelligence.txt  — Early placeholder for final competitor write-up
    Webinar info .txt      — Nuno's April 27–30 live workshop (reference doc)

ATTEMPT 4 — Manual Ad Screenshots (21:05 – 21:33)
  Since the scraper wasn't working, Facebook Ads Library was opened manually.
  Nuno's own ads were screenshotted for reference (nuno 1.jpeg through
  nuno 5.jpeg). His funnel and website PDFs were saved. A visual HTML of his
  profile was built (nuno.html). The goal was to use this as a baseline to
  compare against competitors' ads.
  This worked for Nuno. The problem was doing the same for 20+ candidates.

ATTEMPT 5 — Firecrawl (22:31)
  testcrawl.py was created. This was the Firecrawl experiment.
  Firecrawl was installed, the API was learned, and a test crawl was run.
  It worked — Firecrawl successfully scraped page content from websites.
  However, it could not access the Facebook Ads Library in a meaningful way
  (Facebook blocks headless crawlers and requires authentication), and it
  could not pull live YouTube subscriber counts without additional tooling.
  New skill unlocked. Problem not fully solved.

ATTEMPT 6 — competitor_intelligence.py + Apify attempt (23:06 – 23:14)
  competitor_intelligence.py was the most ambitious script of the day:
  a pipeline to pull competitor YouTube data, check for Facebook ad activity,
  score candidates, and output a ranked CSV.

  Also explored: Apify (screenshot: Apify_fb_ads.png) — a third-party scraping
  platform with Facebook Ads Library actors. It worked better than raw scraping
  but required per-query cost and setup time that wasn't feasible mid-session.

  The competitor_intelligence.py script ran. It produced:
    - screenshots/ folder with 20+ Facebook ads library screenshots
      (Jack Roberts, Liam Ottley, Nick Saraev, Riley Brown, Nate Herk,
       Corbin, Alicia Lyttle, Nick Ponte, Pavlo, Derek Cheung, Ed Hill,
       and many more)
    - competitor_data/ folder with .md files for some candidates
    - competitors_scored.csv — a scored output

  The scored CSV existed. But the scoring logic was inconsistent: some
  candidates were scored on partial data, the Facebook ad detection was
  unreliable (some active advertisers were missed, some inactive ones were
  flagged), and YouTube data was stale or estimated.

  Result: Thousands of lines of Python across the day. Multiple approaches.
  Multiple partial solutions. No clean, trustworthy output.

  The code worked in pieces. It did not work as a whole.

PART 3 — THE PIVOT: MANUAL RESEARCH + GHL AS THE FILTER
--------------------------------------------------------------------------------

At this point, the problem was reframed.

The original question was:
  "How do I write code to find Nuno's competitors?"

The new question was:
  "What do I already KNOW that narrows this down?"

The key insight was GoHighLevel.

Nuno Tavares uses GoHighLevel. His audience uses GoHighLevel. GHL is central
to his business model. Therefore, his most relevant competitors would likely
also be in the GHL ecosystem — either as affiliates, award winners, or
featured partners.

GoHighLevel runs a formal recognition program for high-performing affiliates
and agencies. These are people who:
  (a) have built real businesses using GHL
  (b) are actively marketing those businesses
  (c) have been validated by GHL's own team as high performers

This became the new starting pool.

STEP 1 — GHL Award Winners List
  A list of GHL-recognized creators and affiliates was assembled through
  manual research: GoHighLevel's own website, partner pages, YouTube searches
  for "GoHighLevel award," and community references.

STEP 2 — Claude Segmentation
  With Claude Code, the list was filtered by three criteria simultaneously:
    1. Do they have a YouTube channel? (Must be active, not just existing)
    2. Do they have GHL recognition/awards? (Affiliate, partner, or featured)
    3. Do they cover topics adjacent to Nuno? (AI, marketing, automation,
       agency building, courses — anything in the orbit)

  Dozens of names were processed. Most fell out on criterion 1 or 3.

STEP 3 — Facebook Ads Library Manual Check
  For each candidate who passed the Claude segmentation, the Facebook Ads
  Library (facebook.com/ads/library) was manually searched.
  This is the most reliable way to confirm active paid advertising — no API,
  no scraper, just the source.

  Candidates confirmed running active Facebook/Instagram ads were kept.
  Organic-only creators were removed.

PART 4 — FINAL INTELLIGENCE RUN WITH CLAUDE CODE + RIVALSEARCHMCP
--------------------------------------------------------------------------------

With the final three competitors selected, deep intelligence was gathered using
Claude Code with the RivalSearchMCP tool (web_search, map_website,
research_topic) — the same GitHub repo used throughout this project as the
operational environment.

RivalSearchMCP ran parallel searches across DuckDuckGo and Yahoo, scraped
review sites (marksinsights.com, ippei.com, salesforge.ai, trustpilot.com),
pulled social media profiles, mapped product websites, and cross-referenced
pricing data from Reddit, Scribd, and third-party analytics platforms.

The output was two documents:

  competitor_intelligence_report_2026.txt
  — Full structured report with 7 sections per competitor, pricing tables,
    funnel maps, social media stats, GHL connection analysis, and a
    master comparison table.

  PROCESS_BRIEF.txt (this document)
  — The story of how we got there.

PART 5 — HONEST REFLECTION ON WHAT WOULD HAVE BEEN BETTER
--------------------------------------------------------------------------------

If more time had been available, this is what a better version looks like:

BETTER ALGORITHM (v2 wishlist):
  - Start with GHL affiliate/partner lists as the seed pool, not general YouTube
  - Add Facebook Ads Library check via Apify actor (automated, not manual)
  - Add sales funnel detection: does the candidate have a VSL page? A calendar
    booking page? A Skool community? These are programmatically detectable.
  - Weight the scoring: YouTube subs * 0.2 + active FB ads * 0.4 +
    sales funnel detected * 0.3 + topic overlap * 0.1
  - Output a ranked shortlist of 10–15 candidates, then apply manual review
    only to the top tier

The manual process that was actually used produced accurate results but
would not scale. For a second client or a second assignment, rebuilding the
algorithm with these three filters as hard requirements (YouTube ✓ + FB ads ✓
+ funnel ✓) would turn a 20-hour manual process into a 20-minute automated one.

That is Part 2 of this assignment. Or the next time there is enough time.
